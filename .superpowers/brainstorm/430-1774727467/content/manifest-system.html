Manifest anatomy — annotated

# modules/llm-agent/manifest.toml [module] name = "llm-agent" version = "1.2.0" description = "AI chat, tool-use, and skills system" optional = true # user can skip at setup process = "engine" # which process hosts this module [requires] modules = [] # no module dependencies [requires.hardware] ram_gb_min = 4 # minimum to show in setup gpu_modes = ["cuda", "cpu", "api"] [requires.privilege] level = "user" # no admin needed [python] packages = [ "openai>=1.0", "tiktoken", "httpx", ] packages_gpu = [ # only installed on GPU path "torch", ] [capabilities] llm_adapter = "auto" # ollama → llama.cpp → api [provides] tools = ["llm_chat", "llm_analyze", "llm_stream"] ui = ["ChatBox", "LLMConfig"] wizards = [] # no wizards from this module reports = [] [provides.nav] icon_rail = false # doesn't add a nav icon right_panel_tab = "chat" # adds tab to right panel [safety] max_permission = "read" # tools it registers can read state audit_log = true
[module] block
Identity + which process hosts it. The process orchestrator reads this to know if it needs to start the Engine process at all.
[requires.hardware]
Setup wizard runs capability detection first, then filters the module list. A device with 2GB RAM never sees llm-agent as an option.
[python] packages
Deps are installed per module into the embedded Python. packages_gpu only installs on GPU path — CPU installs skip torch entirely.
[provides] tools
These tool names are registered into the global Tool Registry at startup. Wizards discover tools by name — they never import Python modules directly.
[safety] max_permission
Hard cap on what any tool from this module can do. Prevents a future tool author from accidentally registering a "dangerous" tool under a "read-only" module.

Tool Registry — populated at boot from installed manifests

At startup, each process scans installed.json and loads only its enabled modules. Each module registers its tools into a shared registry. Wizards and the LLM only see tools in this registry.

How setup wizard uses manifests

Hardware detection → module filter

# electron/capability_detector.py # Runs ONCE at first launch, result cached caps = HardwareCaps( gpu_vram_gb = detect_gpu_vram(), # 0 if no GPU ram_gb = detect_ram(), npcap = check_npcap(), # Win only libpcap = check_libpcap(), # Mac/Linux os = sys.platform, disk_gb = detect_free_disk(), ) # Filter manifests by hardware caps compatible = [ m for m in load_all_manifests() if caps.satisfies(m.requires.hardware) ] # → only compatible modules shown in wizard

Process orchestrator — boot only what's needed

# electron/main.js — on app start installed = read_installed_json() needs_daemon = installed.modules.any( m => m.enabled && m.process == "daemon" ) needs_engine = installed.modules.any( m => m.enabled && m.process == "engine" ) # Always start gateway spawn("gateway/main.py") if (needs_daemon): spawn_elevated("daemon/main.py") # UAC if (needs_engine): spawn("engine/main.py") # CPU-only user: no GPU modules installed # → Engine never starts (save RAM)

Frontend — feature flags from installed.json

// frontend/src/store/features.ts // Gateway serves /features endpoint from installed.json const features = await api.get("/features") // → { capture: true, modbus: true, // llm-agent: true, ics-dnp3: false } // Store in Zustand — never changes at runtime useFeatureStore.setState(features)
// In any component: const hasModbus = useFeature("modbus") return ( <IconRail> <NavIcon view="capture" /> // always {hasModbus && <NavIcon view="modbus" />} // conditional </IconRail> )
// Lazy-load feature panel only if installed const ModbusPanel = React.lazy( () => import("../features/modbus/ModbusPanel") ) // In route/view map: const VIEWS = { capture: () => <PacketsAndInsights />, modbus: () => hasModbus ? <Suspense><ModbusPanel /></Suspense> : <NotInstalled module="modbus" />, // ics, reporting, etc. same pattern } // NotInstalled shows: // "This module isn't installed. // Run setup to add it."

Dependency resolution — at install time, not runtime

Example: user selects "reporting"

  • reporting requires llm-agent
  • llm-agent requires nothing
  • Setup wizard auto-selects llm-agent as a dependency
  • User sees: "Reporting requires LLM Agent (auto-added)"
  • If user deselects llm-agent, reporting is also deselected

Adding a module later (no reinstall)

  • User goes to Settings → Modules
  • Sees uninstalled compatible modules
  • Clicks "Install ICS/DNP3"
  • Pip installs only that module's deps
  • installed.json updated
  • Gateway sends module.loaded event over Redis
  • All processes hot-reload that module — no restart needed

What happens on unsupported device

  • Device has no GPU, 4GB RAM, no Npcap
  • Capability detector sets caps
  • Setup shows: capture = incompatible, modbus = ok, llm-agent = CPU mode
  • User can still install modbus + llm (CPU) — just no packet capture
  • UI hides Capture nav icon entirely
