Process model — only start what's installed

ProcessPrivilegeHosts modulesOptional?
gateway user HTTP/WS server, state, scheduler, channels, net tools no — always
daemon admin capture, modbus, ICS/DNP3, subnet scanner, exec tool yes — if no cap modules installed, skip
engine user LLM agent, RAG, expert analysis, skills, reporting yes — skip if no llm-agent installed
worker user long-running tasks: report gen, batch analysis, wizard runs yes — spawned on demand, exits when idle

Wizard & Report framework — composable from tools

Wizard DSL

  • Each wizard is a sequence of steps
  • Steps call registered tools by name
  • Tools can be local (Python) or LLM-powered
  • Wizard definition is a TOML/JSON file — no code
  • Frontend wizard framework renders steps generically
  • Result feeds directly into reporting engine
# wizards/modbus-health.toml [[steps]] title = "Scan for Modbus devices" tool = "modbus_scan" args = {subnet: "$auto"} [[steps]] title = "Read register baseline" tool = "modbus_read" [[steps]] title = "AI analysis" tool = "llm_analyze" prompt= "Assess health of these Modbus devices" [[steps]] title = "Generate report" tool = "report_generate" template = "modbus-audit"

Safety model

  • Every tool has a permission level: read / write / exec / dangerous
  • Wizards declare max permission they need — user must approve at launch
  • Dangerous tools (exec, modbus write) require explicit unlock per session
  • All tool calls are audit-logged with timestamp, args, result hash
  • Process sandbox: exec tool runs in Daemon (isolated from LLM context)
  • No tool can escalate privilege — privilege is fixed at process level
Rule: LLM never calls privileged tools directly. It sends a tool.request to Daemon over Redis. Daemon validates, executes, and returns sanitized output. LLM context never sees raw exec output.

Frontend module structure

Feature-based folders (not type-based)

frontend/src/ ├── features/ │ ├── capture/ # components + store + api + types │ ├── modbus/ │ ├── llm/ │ ├── rag/ │ ├── topology/ │ └── reporting/ ├── wizards/ │ ├── WizardEngine.tsx # generic runner │ └── definitions/ # TOML-loaded ├── shell/ # layout, nav, icon rail ├── shared/ # Toast, hooks, api client └── store/ # per-feature slices

Performance patterns

  • Lazy import feature modules — React.lazy + Suspense per feature panel
  • Virtual lists — @tanstack/virtual for packet table (5,000+ rows)
  • WebWorker for packet parsing / protocol coloring (off main thread)
  • Zustand slices — each feature owns its store, no global re-renders
  • WS backpressure — packet stream throttled to 30fps in UI
  • Code-split by feature — only load installed modules
