tender-fermi  5 architectural conflicts

Built on the OLD monolithic backend. New 3-process design uses Redis streams between Gateway → Engine → Daemon. These conflicts are where the two models clash:

Feature tender-fermi does 3-process expects Fix needed
🔴 Autonomous/Shell mode Module-level globals in backend/agent/chat.py Engine + Gateway are separate processes — globals don't sync Store flags in Redis config stream, read on each process start
🔴 Exec tool Runs in Engine (via symlink to backend/agent/tools/exec.py) Shell execution = privileged → Daemon's job Move exec.py to Daemon; Engine sends ns:tool.request
🟠 Background tasks Adds backend/agent/tasks.py, API endpoints in Gateway legacy routes Long-running tasks belong in Engine, not mounted in Gateway Run task loop in Engine; Gateway proxies via ns:task.request
🟠 Topology endpoints Adds /topology + /topology/scan in backend/api/routes.py; calls tshark directly tshark = privileged = Daemon's job, not Gateway's Move topology generation to Daemon; Gateway sends ns:tool.request
🟡 Tool auth Passes allow_dangerous flag from module globals Multi-process = flags must travel in the Redis message, not via globals Embed allow_dangerous in ns:tool.request payload
